## Concept Understood Today

**Concept name:**  Why SQLite window functions matter for audit
**In my own words:**  **`RANK() OVER (PARTITION BY vendor_name ORDER BY amount DESC)`** → Ranks every invoice within each vendor by amount. Invoice ranked #1 per vendor = potential overbilling flag.
**`AVG(amount) OVER (PARTITION BY vendor_name)`** → Gives you the vendor-level average _next to every row_, not in a collapsed group. So you can compute `amount - AVG(amount)` = deviation per invoice. This is your first statistical audit signal.
**`LAG(invoice_date) OVER (PARTITION BY vendor_name ORDER BY invoice_date)`** → Returns the _previous_ invoice date for the same vendor. `invoice_date - LAG(invoice_date)` = days between consecutive invoices. Flag any vendor with < 7 days between invoices as potential duplicate/fraud pattern.

**AP domain analogy:**  
Because when we eventually hand a chunk of text to an LLM agent, we want to have already pre-flagged the high-risk rows using deterministic SQL.

**How this will be used in the final system:**

The LLM handles ambiguity; SQL handles math. Architecturally, this is called **hybrid retrieval** + **rule-based pre-filtering** — a pattern we'll be using constantly.
